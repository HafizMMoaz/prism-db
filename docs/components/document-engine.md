# Component: Document Engine

**Crate:** `prism-doc`
**Status:** Accepted
**Last updated:** 2026-05-15

## Purpose

The document engine is the access method for schemaless JSON-like documents. Each collection is a heap of records whose payloads are self-describing tagged binary documents. Indexes can be declared on document field paths.

The engine implements CRUD with a MongoDB-subset query language because that language is familiar to the audience.

## Document format

Documents are tagged binary, similar in spirit to BSON but our own format. Detailed bytes in `specs/record-format.md`. Conceptually:

```
Document := {
  length: u32,
  field_count: u16,
  fields: Field[field_count]
}

Field := {
  type_tag: u8,
  name_len: u16,
  name: bytes[name_len],
  value: depends on type_tag
}
```

Type tags include:
- `Null`, `Bool`, `Int32`, `Int64`, `Double`
- `String` (length-prefixed UTF-8)
- `Binary` (length-prefixed bytes + subtype)
- `Array` (length-prefixed; recursively documents)
- `Object` (length-prefixed; recursively documents)
- `Timestamp` (i64 microseconds since epoch)
- `ObjectId` (12 bytes — random, monotonic)

Every document has a top-level `_id` field. Inserts without one get an `ObjectId` generated by the server.

## Public interface

```rust
pub struct DocCollection { /* opaque */ }

impl DocCollection {
    pub fn insert_one(&self, txn: &TxnHandle, doc: Document) -> Result<ObjectId>;
    pub fn insert_many(&self, txn: &TxnHandle, docs: Vec<Document>) -> Result<Vec<ObjectId>>;
    pub fn find_one(&self, txn: &TxnHandle, query: &Query) -> Result<Option<Document>>;
    pub fn find(&self, txn: &TxnHandle, query: &Query) -> Result<DocIterator>;
    pub fn update_one(&self, txn: &TxnHandle, query: &Query, update: &Update) -> Result<u64>;
    pub fn update_many(&self, txn: &TxnHandle, query: &Query, update: &Update) -> Result<u64>;
    pub fn delete_one(&self, txn: &TxnHandle, query: &Query) -> Result<u64>;
    pub fn delete_many(&self, txn: &TxnHandle, query: &Query) -> Result<u64>;
}
```

## Query language

A subset of MongoDB's query operators:

```json
{
  "field": "value",                        // equality
  "field": { "$eq": "value" },             // explicit equality
  "field": { "$gt": 5 },                   // comparison
  "field": { "$lt": 5 },
  "field": { "$gte": 5 },
  "field": { "$lte": 5 },
  "field": { "$ne": "value" },
  "field": { "$in": ["a", "b"] },
  "field": { "$nin": ["a"] },
  "field": { "$exists": true },
  "$and": [ ... ],
  "$or": [ ... ],
  "$not": { ... }
}
```

Nested fields are addressed by dotted path: `address.city`. Array element access: `tags.0` for index, `tags` for any-element predicates.

What we do **not** support in v1:
- `$where` (server-side JavaScript)
- `$regex`
- `$elemMatch` (array-of-objects matching)
- Geospatial operators
- Text search operators
- Aggregation pipeline (`$group`, `$lookup`, etc.) — use the SQL engine for analytical-ish workloads

## Update operators

```json
{
  "$set": { "field": "value" },
  "$unset": { "field": "" },
  "$inc": { "counter": 1 },
  "$push": { "tags": "new" },
  "$pull": { "tags": "old" }
}
```

Updates are applied tuple-at-a-time. The engine reads the existing document, applies the update operators in order, serializes the result, and calls `record_store.update`.

## Query compilation

```
Query JSON
  ↓ parse
Query AST
  ↓ bind (resolve field paths)
Bound query
  ↓ plan (choose index or scan)
Physical plan: IndexLookup | SeqScan + Filter
  ↓ execute
Document iterator
```

### Index selection
For each indexed field path declared on the collection:
- If the query has an equality predicate on that path → use IndexLookup with key = predicate value.
- If the query has a range predicate (`$gt`, `$lt`, etc.) on a B+tree-indexed path → use IndexLookup with range.
- Otherwise → SeqScan + Filter.

We use the most selective available index. "Most selective" is heuristic in v1: equality > range > no match. Future versions add statistics.

### Field path indexes
A user can declare:
```
db.events.createIndex({ "user_id": 1 })   // ascending B+tree
```

This creates a B+tree index whose keys are the serialized value of the document's `user_id` field. The index entry's value is the RID of the document.

When the field doesn't exist on a given document, the document is not indexed (consistent with MongoDB). Indexes can have `sparse: true` declared (it's the default for field-path indexes; documents missing the field are simply absent from the index).

When the field is an array, each array element produces an index entry pointing at the same RID (multikey index, in MongoDB parlance). v1 supports this for B+tree only.

## Filter evaluation

For scan-and-filter execution, the filter is compiled to a tree of evaluators:

```rust
enum FilterNode {
    Eq(FieldPath, Value),
    Gt(FieldPath, Value),
    ...
    And(Vec<FilterNode>),
    Or(Vec<FilterNode>),
}

fn eval(node: &FilterNode, doc: &Document) -> bool;
```

Field paths are parsed once at query compile time. Documents are walked field-by-field when evaluating.

## Type comparison

MongoDB has a defined cross-type comparison order: Null < Number < String < Object < Array < Binary < ObjectId < Bool < Timestamp. We follow this. Within numeric types (Int32, Int64, Double), comparisons are numeric, not type-strict.

## Concurrency

Documents are records; MVCC and lock manager apply. Concurrent updates to the same document follow snapshot isolation semantics. Cross-document operations have no special semantics — they are not atomic across documents unless wrapped in a transaction.

## Recovery

Documents are records in pages. Every insert, update, delete produces an Insert/Update/Delete WAL record. Recovery replays them. Index updates likewise.

## Configuration

```toml
[doc_engine]
default_objectid_strategy = "monotonic_random"
max_document_size_bytes = 4194304        # 4 MiB; documents larger rejected at insert
```

Document size cap is below page size minus header overhead. Larger documents would need overflow page support, post-v1.

## Testing

- Unit: query parser, filter evaluator, update operator application.
- Property: arbitrary documents and queries; compare results against a reference (a Python dict-based oracle).
- Index correctness: queries with and without indexes return the same results.
- Recovery: documents inserted before crash recover atomically.
